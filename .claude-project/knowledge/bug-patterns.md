## Category: 인증/로그인

### Pattern: 세션 만료/자격증명 오류 반복 노출
- **빈도**: 5회 (높음)
- **증상**: 사용자가 정상 사용 중 자격증명 오류 팝업이 반복 노출됨. 페이지를 새로고침하면 해결되지만 빈번히 재발생
- **근본 원인**: 서버 세션 제한 또는 토큰 갱신 실패 시 클라이언트에서 적절한 핸들링 없이 에러를 그대로 노출
- **영향 화면**: 대시보드, 드롭다운 메뉴, 홈 화면
- **예방 스펙**: "[클라이언트] 세션 만료(401/403) 응답 수신 시 자동으로 토큰 갱신을 시도하고, 갱신 실패 시 로그인 페이지로 리다이렉트하며 사용자에게 '세션이 만료되었습니다' 메시지를 1회만 표시한다. 동일 에러를 중복 노출하지 않도록 에러 상태를 전역으로 관리한다."
- **사례**: [#1252, #1251, #1244, #1240, #1576]

### Pattern: 로그인/회원가입 동작 불가
- **빈도**: 10회 (높음)
- **증상**: 로그인 버튼이 반응하지 않음. 회원가입이 작동하지 않음. 가입 중 멈춤. 로그아웃 시 401 에러 발생.
- **근본 원인**: 폼 유효성 검사 로직 오류로 인한 버튼 비활성화 유지. API 엔드포인트 오류 또는 토큰 처리 실패. 로그아웃 API 호출 시 만료된 토큰 전송.
- **영향 화면**: 로그인 화면, 회원가입 화면, 로그아웃 처리
- **예방 스펙**: "[클라이언트] 로그인/회원가입 버튼은 폼 유효성 검사 통과 여부와 무관하게 클릭 가능하도록 하고, 검사 실패 시 각 필드에 구체적인 오류 메시지를 표시한다. [서버] 로그아웃 API는 토큰 만료 여부와 관계없이 성공 응답을 반환하고 서버 측 토큰을 무효화한다."
- **사례**: [Coaching Record #Login button not responding, K Talk #517 Sign Up is not working, K Talk #stuck at signup, K Talk #Login issue in client Server-site, #1064 User log out error, Trustix #2000, Trustix #1999, Trustix #1998, Trustix #1972, Trustix #1971]

### Pattern: 인증 후 잘못된 화면으로 리다이렉트
- **빈도**: 4회 (중간)
- **증상**: 로그인 완료 후 대시보드가 아닌 비밀번호 재설정 완료 화면, 로그인 페이지 등 잘못된 화면으로 이동. 비인증 사용자가 홈 화면에 접근 가능
- **근본 원인**: 인증 상태 체크 및 라우팅 가드 로직 미적용 또는 네비게이션 스택 관리 오류
- **영향 화면**: 로그인 화면, 회원가입 완료 화면, 홈 화면
- **예방 스펙**: "[클라이언트] 모든 라우트에 인증 가드를 적용한다. 미인증 사용자의 보호된 라우트 접근 시 로그인 페이지로 리다이렉트한다. 인증된 사용자가 로그인/온보딩 페이지 접근 시 대시보드/홈으로 리다이렉트한다. 회원가입 완료 후 네비게이션 스택을 초기화하여 뒤로가기로 인증 전 화면에 접근할 수 없도록 한다."
- **사례**: [#2226, #2234, #2928, #2894]

### Pattern: 비밀번호 유효성 검사 미흡
- **빈도**: 6회 (높음)
- **증상**: 규칙(8-12자, 문자+숫자+특수문자)에 맞지 않는 비밀번호로 회원가입 가능. 비밀번호 변경 성공 후 404 에러 표시. 잘못된 비밀번호 입력 시 에러 메시지 미흡
- **근본 원인**: 클라이언트 측 유효성 검사 누락 또는 서버 응답 후 성공/실패 분기 처리 오류
- **영향 화면**: 회원가입, 비밀번호 변경, 로그인
- **예방 스펙**: "[클라이언트] 비밀번호 입력 필드에 실시간 유효성 검사를 적용하고, 규칙 미충족 시 구체적인 안내 메시지를 표시한다. [서버] 비밀번호 변경 API는 성공(200) 시 성공 응답을, 실패 시 적절한 에러 코드를 반환한다. [클라이언트] 잘못된 비밀번호 입력 시 '아이디 또는 비밀번호가 일치하지 않습니다' 메시지를 표시한다."
- **사례**: [#2273, #1272, #2932, #2235, #2240, Trustix #1996, Trustix #1991, Bside #3015]

### Pattern: 동일 계정 다중 기기 로그인 처리 오류
- **빈도**: 3회 (중간)
- **증상**: 이미 다른 기기에서 로그인 중임에도 동일 계정으로 재로그인 가능하거나, 반대로 정상 사용 중 "다른 기기에서 이용 중" 메시지가 반복 표시됨
- **근본 원인**: 단일 기기 로그인 강제 로직의 세션 무효화 처리 미흡 또는 락 해제 타이밍 불일치
- **영향 화면**: 로그인 화면, 홈 화면
- **예방 스펙**: "[서버] 로그인 시 기존 세션을 즉시 무효화하고 새 세션을 발급한다. [클라이언트] 다중 기기 로그인 감지 시 명확한 안내 후 기존 세션을 강제 종료하는 옵션을 제공한다. 로그아웃 시 서버 세션을 완전히 삭제하여 재로그인 시 오탐이 발생하지 않도록 한다."
- **사례**: [#2225, #2329, #1575]

### Pattern: 온보딩 화면 반복 노출
- **빈도**: 3회 (중간)
- **증상**: 이미 온보딩을 완료한 사용자에게 앱 시작 시 또는 로그아웃 후 재로그인 시 온보딩 화면이 반복 표시됨
- **근본 원인**: 온보딩 완료 상태를 로컬 스토리지에 저장하지 않거나 초기화 시 상태 확인 로직 누락
- **영향 화면**: 온보딩, 스플래시 화면
- **예방 스펙**: "[클라이언트] 온보딩 완료 여부를 로컬 스토리지(AsyncStorage/SecureStore)에 저장하고, 앱 시작 시 이 값을 확인하여 온보딩 건너뛰기를 결정한다. 로그아웃 시 온보딩 완료 플래그는 초기화하지 않는다."
- **사례**: [#2946, #2954, #2887]

### Pattern: 소셜 로그인 실패 및 플로우 미최적화
- **빈도**: 3회 (중간)
- **증상**: Apple 계정 로그인이 동작하지 않음. 인증 프로세스 마지막 단계에서 서버 오류 발생. 소셜 회원가입 시 중간 화면 누락. 이미 가입된 이메일로 재가입 가능.
- **근본 원인**: 소셜 로그인 콜백 처리 또는 서버 측 토큰 검증 로직 오류. OAuth 콜백 처리 후 세션 복원 및 리다이렉트 로직 미비. 회원가입 시 이메일 중복 검사 로직 누락.
- **영향 화면**: 로그인 화면, 소셜 회원가입 플로우, 이메일 중복 확인
- **예방 스펙**: "[서버] 소셜 로그인(Apple, Google 등) 콜백 처리 시 토큰 검증 실패에 대한 명확한 에러 응답을 반환한다. 회원가입 API는 이메일/소셜 계정 중복 여부를 먼저 확인하고, 기존 사용자에게는 '이미 가입된 계정입니다' 메시지와 함께 로그인 화면으로 안내한다. OAuth 콜백 핸들러는 토큰 발급 후 즉시 클라이언트 딥링크로 리다이렉트해야 한다. [클라이언트] 소셜 로그인 실패 시 사용자에게 재시도 안내 메시지를 표시하고 일반 로그인 옵션을 제공한다."
- **사례**: [#2243, #2935, #3082, Coaching Record #Social signup one screen missing, Ivory #If existed user tries to signup]

### Pattern: 회원가입 폼 — API 미연결
- **빈도**: 2회 (낮음)
- **증상**: 회원가입 폼 제출 시 계정이 생성되지 않거나, 특정 필드(라이선스 번호, 병원 코드)가 저장되지 않음
- **근본 원인**: handleSubmit에서 실제 API 호출 없이 navigate()만 실행하거나, 백엔드 DTO/Entity에 필드가 누락되어 전체 파이프라인이 불완전
- **영향 화면**: 코치 회원가입 폼
- **예방 스펙**: "[클라이언트] 회원가입 폼 제출 핸들러는 반드시 인증 API를 호출해야 하며, 로딩 상태·에러 핸들링·성공 메시지 UI를 포함해야 한다. [서버] RegisterDto, Entity, Service 세 레이어 모두에 동일 필드가 존재하는지 PR 체크리스트로 검증한다."
- **사례**: [#3070, #3071]

### Pattern: 신규 가입 후 로그인 루프
- **빈도**: 1회 (낮음)
- **증상**: 신규 가입 직후 로그인 시 등록 페이지로 계속 리다이렉트되는 무한 루프
- **근본 원인**: 가입 완료 후 인증 상태가 갱신되지 않아 라우팅 가드가 미인증으로 판단하고 등록 페이지로 반복 전송
- **영향 화면**: 신규 사용자 로그인 플로우
- **예방 스펙**: "[클라이언트] 회원가입 성공 후 인증 토큰/상태를 즉시 갱신하고, 라우팅 가드가 새 상태를 참조하도록 한다. 가입→로그인→홈 플로우를 E2E 테스트로 커버한다."
- **사례**: [#1283]

### Pattern: 로그인 실패 메시지 미표시
- **빈도**: 1회 (낮음)
- **증상**: 잘못된 비밀번호 입력 시 아무런 오류 메시지가 나타나지 않음
- **근본 원인**: 로그인 실패 시 에러 상태를 UI에 바인딩하지 않음
- **영향 화면**: 관리자 로그인 페이지
- **예방 스펙**: "[클라이언트] 로그인 실패 응답(401/403) 수신 시 반드시 인라인 오류 메시지를 노출해야 한다. 에러 메시지는 i18n 키를 통해 다국어를 지원한다."
- **사례**: [#1531]

## Category: 이미지/파일 업로드

### Pattern: 프로필 이미지 변경 실패 및 앱 크래시
- **빈도**: 3회 (중간)
- **증상**: 프로필 이미지 변경 후 반영되지 않음. 카메라로 프로필 사진 변경 시 앱이 크래시됨(iOS/Android 모두 발생).
- **근본 원인**: 카메라 권한 처리 미흡, 이미지 압축/리사이징 처리 중 메모리 오류, 업로드 후 UI 캐시 갱신 누락
- **영향 화면**: 프로필 편집 화면, 이미지 선택기
- **예방 스펙**: "[클라이언트] 카메라/갤러리 접근 전 권한 요청 플로우를 반드시 구현하고, 권한 거부 시 설정 안내 팝업을 표시한다. 이미지 업로드 전 최대 2MB로 압축하고, 업로드 성공 후 즉시 로컬 캐시를 무효화하여 새 이미지를 표시한다. [서버] 이미지 업로드 완료 응답에 새 이미지 URL을 포함하여 반환한다."
- **사례**: [K Talk #Profile image didn't change, Coaching Record #Changing profile picture on iphone/Android by camera crashes app]

### Pattern: 파일 업로드 실패 및 요구사항 오류
- **빈도**: 5회 (높음)
- **증상**: 파일 업로드가 실패함. 업로드 파일 요구사항(크기, 포맷)이 명확하지 않거나 잘못 표시됨. STL 파일 업로드가 작동하지 않음.
- **근본 원인**: 파일 타입/크기 유효성 검사 로직 누락 또는 서버 측 파일 처리 오류, 에러 메시지 미표시
- **영향 화면**: 파일 업로드 화면, STL 파일 업로드, 주문 첨부 파일
- **예방 스펙**: "[클라이언트] 업로드 전 허용 파일 형식과 최대 크기를 명확히 안내하고, 조건 미충족 시 구체적인 에러 메시지를 표시한다. [서버] 파일 업로드 API는 허용 MIME 타입 화이트리스트와 최대 파일 크기를 서버 측에서도 검증하고, 실패 시 명확한 오류 코드를 반환한다."
- **사례**: [Elite4Print #Upload New File Issues, Elite4Print #Upload File Requirement Issue, Ivory #Upload approved designed STL not working]

### Pattern: 업로드된 이미지 채팅 UI 미표시
- **빈도**: 1회 (낮음)
- **증상**: 이미지를 전송했음에도 채팅 UI에 이미지가 표시되지 않음
- **근본 원인**: 이미지 데이터 URL이 백엔드로는 전송되나 Redux 상태 및 MessageBubble 컴포넌트에 바인딩되지 않음
- **영향 화면**: 채팅 상세 페이지 (이미지 첨부)
- **예방 스펙**: "[클라이언트] 파일/이미지 전송 후 해당 메시지 객체에 미리보기 URL을 포함해 즉시 렌더링해야 한다. 업로드 성공 여부와 관계없이 낙관적 UI로 먼저 표시한다."
- **사례**: [#3081]

### Pattern: STL 파일 삭제 후 업로드 흐름 중단
- **빈도**: 1회 (낮음)
- **증상**: 업로드된 STL 파일 삭제 시 "Skip for now" 버튼이 동작하지 않아 재업로드 없이 진행 불가
- **근본 원인**: 파일 삭제 후 업로드 상태 초기화가 완전하지 않아 다음 단계 진행 조건이 잘못 평가됨
- **영향 화면**: STL 파일 업로드, 주문 편집 페이지
- **예방 스펙**: "[클라이언트] 파일 삭제 시 업로드 상태를 완전히 초기화(null/empty)하고, 필수 파일 없이도 '건너뛰기'가 가능한 경우 해당 조건을 명시적으로 처리한다."
- **사례**: [#1279]

### Pattern: 갤러리 접근 권한 미요청
- **빈도**: 1회 (낮음)

## Category: 리스트/페이지네이션

### Pattern: 페이지네이션 미구현 및 정렬 오류
- **빈도**: 6회 (높음)
- **증상**: 목록 페이지에 페이지네이션이 없어 모든 데이터가 한 번에 표시됨. 최신 항목이 목록 상단에 표시되지 않음. 페이지 번호가 제대로 동작하지 않음. 설문 히스토리가 10개만 표시됨. 주문 정보 테이블에 일부 주문 미표시.
- **근본 원인**: 페이지네이션 API 파라미터 미적용, 기본 정렬 조건 미설정(created_at DESC 누락), UI 컴포넌트 미구현. useHistory 훅이 페이지네이션 파라미터를 지원하지 않고 고정 limit=10으로 조회.
- **영향 화면**: 사용자 목록, 블로그 목록, 어드민 패널 목록, 주문 목록, 환자 설문 히스토리 탭, 관리자 사용자 상세 주문 정보 섹션
- **예방 스펙**: "[서버] 목록 조회 API는 반드시 page, page_size 파라미터를 지원하고 기본 정렬은 created_at DESC로 설정한다. 리스트 API는 total, page, limit을 포함한 페이지네이션 응답 구조를 반환한다. 사용자 주문 목록 API는 상태 필터 없이 전체 주문 수(totalCount)를 별도 반환해야 한다. [클라이언트] 10개 이상의 데이터가 예상되는 모든 목록 화면에 페이지네이션 또는 무한 스크롤을 구현하며, 빈 상태(empty state) UI도 함께 제공한다. 테이블 상단 카운트와 실제 렌더링 행 수가 불일치하면 경고 로그를 출력한다."
- **사례**: [Near Work #Need a pagination system, Thrll #website blog page don't have pagination, Onui CRM #need Sorting by created at, Onui CRM #New user should be display top of the list, #1530, #1542]

### Pattern: 필터 적용 후 자동 초기화
- **빈도**: 5회 (높음)
- **증상**: 필터 모달을 닫고 다시 열면 설정한 필터(진단, 날짜, 통화 결과)가 초기화됨. 필터 적용 자체가 동작하지 않음.
- **근본 원인**: 필터 상태가 모달 컴포넌트 내부 로컬 state로만 관리되어 닫힘 시 소멸. 필터 파라미터가 API 요청에 포함되지 않음.
- **영향 화면**: 콜로그 관리 필터, 리드 목록 필터, 투데이 어사인먼트 필터, 플레이북 필터
- **예방 스펙**: "[클라이언트] 필터 조건은 상위 컴포넌트 또는 전역 상태에 보관하여 모달 재오픈 시 이전 선택값이 유지되도록 한다. 필터 적용 버튼 클릭 시 해당 조건을 API 요청 파라미터에 포함하고, 필터 초기화 버튼을 별도 제공한다."

### Pattern: 뒤로가기 시 페이지/검색 상태 초기화

- **근본 원인**: 목록 페이지 상태가 컴포넌트 로컬 state로만 관리되어 언마운트 시 소멸. 전역 상태 또는 URL query param 활용 미흡.
- **영향 화면**: 주문 목록, 검색 결과 목록
- **예방 스펙**: "[클라이언트] 목록 페이지의 현재 페이지 번호, 검색어, 필터 조건은 URL query parameter 또는 전역 상태 관리(Redux, Zustand 등)에 저장하여 뒤로가기 후 복원되도록 구현한다."
- **사례**: [Ivory #Pagination and search state reset after navigating back, Ivory #1160 Issue on order details after data is revised with back buttons]

### Pattern: 관리자 테이블 Export 버튼 미동작

- **예방 스펙**: "[클라이언트] 버튼 컴포넌트에 onClick 핸들러가 없으면 비활성(disabled) 상태로 표시하거나, 구현 전까지 버튼 자체를 숨겨야 한다. Export 기능은 CSV/Excel 다운로드 API 연결 후 활성화한다."

## Category: 푸시 알림

### Pattern: 알림 뱃지/카운트 오류
- **빈도**: 8회 (높음)
- **증상**: 읽지 않은 알림이 없는데도 빨간 점(red dot)이 표시됨. 알림 숫자가 잘려 보이거나 정적인 값으로 고정됨. 채팅 알림 카운트가 읽음 처리 후에도 초기화되지 않음. 새 채팅 메시지가 있어도 채팅 아이콘에 알림 뱃지가 표시되지 않음.
- **근본 원인**: 알림 카운트 업데이트 API 호출 누락 또는 로컬 상태와 서버 상태 간 동기화 미흡. 뱃지 숫자 UI 컴포넌트의 최대 표시 길이 미처리. 미읽음 메시지 카운트가 채팅 아이콘 컴포넌트에 전달되지 않거나, 알림 이벤트 구독이 누락.
- **영향 화면**: 알림 탭 뱃지, 채팅 알림 카운트, 홈 화면 아이콘 뱃지, 채팅 아이콘, 알림 센터
- **예방 스펙**: "[클라이언트] 알림 목록 진입 시 서버에서 unread count를 재조회하여 로컬 상태를 동기화한다. 뱃지 숫자는 99+처럼 오버플로우 처리 필수. 채팅 아이콘은 미읽음 메시지 수를 실시간으로 반영하는 배지를 표시해야 한다. 소켓 수신 이벤트 또는 주기적 폴링으로 미읽음 수를 갱신한다. [서버] '모두 읽음' 처리 API는 실제 DB 업데이트 후 count=0 응답을 보장해야 하며, 클라이언트는 응답 수신 후 즉시 뱃지를 갱신한다."
- **사례**: [Ivory #No new notification but showing red dot, Coaching Record #Notification number cut and static, Coaching Record #Chat notification count number not clearing, #1365, #1363]

### Pattern: 알림 미수신 및 백그라운드 알림 불가
- **빈도**: 5회 (높음)
- **증상**: 앱이 꺼져 있거나 백그라운드 상태일 때 푸시 알림이 도달하지 않음. 알림을 아예 받지 못하는 경우도 발생.
- **근본 원인**: 푸시 알림 전송 로직이 앱 포그라운드 상태에만 의존하거나, FCM/APNs 토큰 갱신 및 백그라운드 처리 설정 누락
- **영향 화면**: 알림 수신 전반, 푸시 알림 설정
- **예방 스펙**: "[클라이언트] FCM/APNs 백그라운드 모드(background fetch, remote notifications)를 반드시 활성화하고, 앱 종료/백그라운드 상태에서도 silent push 및 data payload 방식으로 알림이 도달하도록 구현한다. [서버] 알림 전송 시 priority=high 설정, TTL(time-to-live) 지정 필수. 알림 전송 실패 시 재시도 로직 포함."
- **사례**: [Roadmap #Push notifications should be delivered regardless of phone state, K Talk #Notifications are not sent, Near Work #Notification titles are not providing accurate information, K Talk #notification]

### Pattern: 알림 중복 표시 및 페이지 오동작
- **빈도**: 6회 (높음)
- **증상**: 같은 알림이 두 번 표시됨. 알림 클릭 시 반응 없음. 알림 페이지 자체가 동작하지 않음. 회원가입 후 이전 알림 데이터가 노출됨.
- **근본 원인**: 알림 리스너 중복 등록, 딥링크 핸들러 미연결, 신규 가입 사용자에 대한 알림 데이터 초기화 누락
- **영향 화면**: 알림 목록 페이지, 알림 클릭 딥링크, 회원가입 후 알림 초기 상태
- **예방 스펙**: "[클라이언트] 알림 리스너는 컴포넌트 마운트 시 1회만 등록하고 언마운트 시 반드시 해제한다. 알림 클릭 시 딥링크 또는 해당 화면으로 이동하는 핸들러를 필수 구현한다. [서버] 신규 회원 생성 시 알림 데이터를 빈 상태로 초기화한다."
- **사례**: [Ivory #Notification shows two times, K Talk #Notification is not clickable, Ivory #Notification Page Not working, Roadmap #Notification issues(old data)]

### Pattern: 알림 아이콘 위치 및 메시지 정확도 오류
- **빈도**: 3회 (중간)
- **증상**: 알림 아이콘 위치가 잘못됨. 알림 제목/메시지가 부정확하거나 잘못된 정보를 포함. 알림 메시지가 올바르게 표시되지 않음.
- **근본 원인**: 알림 템플릿 변수 바인딩 오류, 알림 아이콘 레이아웃 미정의, 알림 발송 시 컨텍스트 데이터 오삽입
- **영향 화면**: 알림 아이콘 UI, 알림 메시지 본문
- **예방 스펙**: "[서버] 알림 메시지 템플릿에 삽입되는 변수(사용자명, 콘텐츠명 등)는 발송 전 null/empty 여부를 검증한다. [클라이언트] 알림 아이콘은 디자인 시스템의 표준 아이콘을 사용하고, 위치는 Figma 스펙 기준 픽셀 단위로 고정한다."
- **사례**: [K Talk #Notification icon placement, Near Work #Notification titles are not providing accurate information, Coaching Record #Notification message is not Shown properly]

### Pattern: 알림 클릭 시 딥링크 미동작
- **빈도**: 3회 (중간)
- **증상**: 환자가 알림을 클릭해도 해당 화면으로 이동하지 않음. 디자인 파일 추가 알림 클릭 시 리뷰 화면으로 직접 이동하지 않음.
- **근본 원인**: 알림 클릭 핸들러에 내비게이션 로직이 구현되지 않음. 딥링크 처리 로직 미구현 또는 알림 페이로드와 라우팅 연결 누락.
- **영향 화면**: 환자 알림 목록, 알림 목록, 디자인 리뷰 화면
- **예방 스펙**: "[클라이언트] 알림 아이템에는 클릭 시 이동할 대상 경로(deeplink)를 데이터 속성으로 포함하고, 클릭 핸들러에서 해당 경로로 라우팅한다. 푸시 알림 및 인앱 알림 클릭 시 알림 타입에 맞는 화면으로 직접 이동하는 딥링크를 구현한다. 알림 페이로드에 target_screen과 target_id를 포함하여 정확한 라우팅을 보장한다. 딥링크 URL 형식을 알림 타입별로 스펙에 정의한다."

## Category: 결제/인앱구매

### Pattern: 결제 완료 후 UI 미반영 및 히스토리 오류
- **빈도**: 7회 (높음)
- **증상**: 결제 완료 후 성공 메시지 없이 회색 화면 표시. 결제 내역 페이지가 업데이트되지 않음. 결제하지 않았는데 히스토리에 데이터 표시. 대시보드 결제 금액과 결제 내역 불일치.

- **예방 스펙**: "[서버] 결제 완료 웹훅 수신 후 즉시 결제 상태를 DB에 반영하고, 클라이언트 폴링 또는 소켓 이벤트로 결과를 전달한다. 카드 등록과 결제 완료는 별도 이벤트로 분리 처리한다. [클라이언트] 결제 완료 시 성공/실패 메시지를 명확히 표시하고 결제 내역 페이지로 자동 이동한다."
- **사례**: [Roadmap #dashboard payment amount different, Roadmap #payment history still not showing, Coaching Record #1674 Payment success is showing raw message, Bright Pixel #673 Not getting success message after payment, APP #1414 payment issue]

### Pattern: 쿠폰/프로모션 적용 오류 및 가격 계산 오류
- **빈도**: 5회 (중간)
- **증상**: 쿠폰 코드 적용 시 체크아웃 불가. 장바구니 금액이 소수점 3자리까지 표시됨. 배송비/추가 금액이 인보이스에 미반영. 소계와 최종 합계 금액이 일치하지 않음(중복 청구 의심). 결제 화면에서 가격 항목이 표시되지 않음.
- **근본 원인**: 쿠폰 적용 후 최종 금액 계산 로직 오류. 부동소수점 반올림 처리 미흡. 추가 요금 항목이 인보이스 생성 로직에 포함되지 않음. 배송비 등 추가 항목이 소계에 포함된 후 다시 합산되는 이중 계산 로직 오류.
- **영향 화면**: 장바구니, 체크아웃, 인보이스, 가격 견적, 결제 페이지, 주문 상세 페이지
- **예방 스펙**: "[서버] 모든 금액 계산은 정수(최소 단위: 원/센트) 기반으로 처리하여 부동소수점 오류를 방지한다. 쿠폰 적용 후 최종 금액을 서버에서 재계산하여 반환한다. 인보이스 생성 시 배송비, 추가 요금 등 모든 비용 항목을 포함하는지 체크리스트로 검증한다. [클라이언트] 결제 전 화면에서 모든 가격 항목(상품별 금액, 배송비, 할인, 최종 합계)을 명시적으로 표시한다."

## Category: 실시간 데이터

### Pattern: 실시간 메시지/알림 미수신
- **빈도**: 5회 (높음)
- **증상**: DM 메시지가 실시간으로 표시되지 않음. 특정 페이지에서만 채팅 알림 수신. 다른 페이지에서 메시지 수신 시 알림 없음. 결제 알림 미수신. 주문 상태 변경 및 알림이 실시간으로 반영되지 않음.

- **영향 화면**: DM 화면, 홈 화면, 주문 상세, 결제 화면, 주문 관리 대시보드, 알림 센터
- **예방 스펙**: "[클라이언트] 웹소켓/실시간 연결은 앱 레벨에서 전역으로 관리하여 특정 화면에 종속되지 않게 한다. 앱이 포그라운드로 돌아올 때 연결 상태를 확인하고 끊어진 경우 재연결한다. 미읽은 메시지/알림은 배지 형태로 화면 전환과 무관하게 유지한다. [서버] 주문 상태 변경 시 관련 사용자에게 실시간 이벤트를 emit해야 한다. 소켓 연결 해제 시 재연결 및 놓친 이벤트 재조회 로직을 구현한다."

### Pattern: 채팅 UI 오류 및 메시지 오동작
- **빈도**: 4회 (중간)
- **증상**: 채팅 텍스트가 UI 영역을 벗어나 overflow됨. 채팅 상대방 이름이 잘못 표시됨. 관리자가 대시보드에서 보낸 메시지가 채팅에서 'Coach'로 표시됨. 채팅 메시지가 최신 순(내림차순)으로 표시되어 가장 최근 메시지가 상단에 위치.
- **근본 원인**: 채팅 말풍선 최대 너비 CSS 미설정. 채팅 참여자 데이터 바인딩 오류. 발신자 식별을 ID 비교에만 의존하여 관리자가 코치이기도 한 경우 역할 오인식. 백엔드가 페이지네이션을 위해 DESC 정렬로 반환하지만 프론트엔드가 배열을 역순으로 정렬하지 않고 렌더링.
- **영향 화면**: 채팅 화면, 메시지 목록, 환자 채팅 상세, 코치 채팅 상세
- **예방 스펙**: "[클라이언트] 채팅 메시지 말풍선은 최대 너비(max-width: 70-80%)를 설정하고 긴 텍스트는 word-break: break-word를 적용한다. 채팅 참여자 정보는 채팅방 입장 시 서버에서 재조회하여 최신 데이터를 표시한다. 발신자 표시는 ID 비교보다 role 필드를 우선 사용한다. 메시지 렌더링 전 항상 createdAt ASC 기준으로 정렬 후 표시한다. [서버] 채팅 메시지 목록 API는 실제 렌더링 순서(오래된 순 = ASC)를 기본으로 반환하거나, DESC 정렬 시 응답에 명시한다. 소켓 메시지 브로드캐스트 및 채팅방 조회 응답에 sender.role 필드를 포함해야 한다."
- **사례**: [K Talk #Chat Page, Ivory #chat text running over, Coaching Record #Chat restricted. Chat person name incorrect., #2977, #2978]

- **예방 스펙**: "[서버] 채팅방 생성 엔드포인트는 getOrCreate 패턴을 사용하고, DB UNIQUE 제약 + 레이스 컨디션 처리(23505 에러 캐치)를 구현해야 한다. 중복 채팅방이 존재할 경우 마이그레이션으로 병합한다."

### Pattern: 온라인/오프라인 상태 표시 오류

### Pattern: 어드민 필터/통계 오류
- **빈도**: 6회 (높음)

- **근본 원인**: 필터 API 파라미터 기본값 오설정. 통계 쿼리에서 전체 상태 값 미포함. 권한 체크 로직 미구현. 통계 데이터를 실시간 또는 주기적으로 갱신하지 않고 초기 로드 데이터만 표시.
- **영향 화면**: 주문 목록 필터, MQL 통계, 권한 관리, 콜로그 관리, 관리자 리포트, 수입 대시보드
- **예방 스펙**: "[서버] '전체' 필터 선택 시 status 파라미터를 생략하거나 all 값으로 전체 데이터를 반환한다. 통계 API는 모든 상태값을 집계에 포함하고, 값이 0인 상태도 응답에 포함한다. 결제 완료 이벤트 발생 시 관련 통계를 즉시 갱신한다. [클라이언트] 권한 설정 저장 후 해당 권한을 즉시 적용하고 재로그인 없이 반영되는지 검증한다. 통계/수입 데이터는 화면 포커스 시 재조회하거나 일정 주기(예: 30초)로 자동 갱신한다."
- **사례**: [Ivory #All Filter in Order List Displays Only Pending Status, Onui CRM #MQL - status, Onui CRM #Permission management, #1257, #754]

### Pattern: 관리자 벌크/단일 액션 오류

- **영향 화면**: 사용자 관리, 어드민 관리, 주문 관리
- **예방 스펙**: "[서버] 어드민 전용 API 엔드포인트는 role 기반 접근 제어(RBAC)를 적용하고, superadmin이 admin을 생성/삭제할 수 있는 권한 체계를 명확히 정의한다. 모든 상태 전환(Hold 포함)에 대한 API를 사전 구현하고 테스트한다."

### Pattern: 관리자 사용자 업데이트 DTO 필드 불일치
- **빈도**: 2회 (낮음)
- **증상**: 관리자가 환자 정보(fullName, phone, birth) 또는 코치 배정(coachId)을 저장하려 하면 400 오류 발생

- **예방 스펙**: "[서버] UpdateUserDto는 클라이언트가 전송 가능한 모든 수정 필드를 @IsOptional()로 포함해야 한다. DTO 변경 시 프론트엔드 타입 정의도 동시에 업데이트한다."
- **사례**: [#2940, #2921]

### Pattern: 관리자 페이지네이션 상태 유지 실패

### Pattern: 딥링크/라우팅 오류 및 루프

- **예방 스펙**: "[클라이언트] 다단계 플로우(주문, 온보딩 등)의 각 단계는 고유한 라우트를 가지며, 완료 후 히스토리 스택을 초기화(replace 방식)하여 뒤로가기로 완료 단계에 재진입하지 못하도록 한다. 라우팅 인터셉터에서 '#'으로 시작하는 href는 인페이지 스크롤로 처리하고 내비게이션에서 제외한다. 링크 매칭 시 공백을 하이픈으로 정규화하고 소문자로 통일한다."

### Pattern: 타임존 불일치로 인한 날짜 오프셋

- **영향 화면**: 환자 홈(캘린더), 운동 로그, 설문 상태 표시
- **예방 스펙**: "[클라이언트] 날짜 문자열(YYYY-MM-DD)을 Date 객체로 변환할 때 반드시 로컬 타임존 기준 파싱(로컬 자정)을 사용하고 toISOString() 대신 직접 문자열 포맷(YYYY-MM-DD)을 유지한다. today 기준 날짜는 useMemo로 안정화한다. [서버] 날짜 범위 조회 시 start-of-day(00:00:00.000) ~ end-of-day(23:59:59.999) 정규화를 DB 쿼리에 적용한다."

### Pattern: 잘못된 데이터 표시 (Stale/Incorrect Data)

- **영향 화면**: 태스크 상세, 리드 상세, 진단 제출

- **예방 스펙**: "[클라이언트] 데이터 제출 성공 후 관련 쿼리 키를 즉시 invalidate해야 한다. 날짜 비교는 문자열 기반(YYYY-MM-DD) 방식을 사용한다."

### Pattern: 운동 카운트/완료 상태 불일치

### Pattern: 폼 유효성 검사 미동작 및 버튼 비정상 활성화

- **예방 스펙**: "[클라이언트] 모든 입력 필드는 실시간 유효성 검사를 적용하고, 각 오류 유형에 맞는 구체적인 안내 메시지를 표시한다. 전화번호 필드는 허용 자릿수와 포맷(예: 010-XXXX-XXXX)을 자동 마스킹 또는 입력 안내 placeholder로 제공한다. Continue/Submit 버튼은 모든 필수 필드가 유효한 경우에만 활성화한다. 각 필드의 유효성 상태를 개별적으로 추적하고 AND 조건으로 버튼 활성화 여부를 결정한다."

### Pattern: 다단계 폼 데이터 유지 오류 및 루프
- **빈도**: 4회 (중간)
- **증상**: 뒤로가기 버튼 사용 시 이전에 입력한 데이터가 초기화되거나 이전 데이터로 덮어써짐. 주문 수정 시 변경사항이 반영되지 않음. 주문 편집 중 뒤로가기 시 이미 지나온 페이지로 반복 이동하거나, 모든 치아를 크라운으로 선택 후 진행이 멈춤.
- **근본 원인**: 다단계 폼 상태가 각 단계 컴포넌트 로컬 state로만 관리되어 뒤로가기 시 소멸. 다단계 폼의 스텝 상태 관리 로직이 편집 모드에서 조건을 잘못 평가하거나, 완료 조건이 초기 생성과 편집에서 다르게 처리됨.
- **영향 화면**: 주문 생성/수정 다단계 폼, 주문 생성/편집 폼 (Abutment, Crown 단계)
- **예방 스펙**: "[클라이언트] 다단계 폼의 입력 데이터는 전역 상태 또는 폼 컨텍스트에 저장하여 단계 간 이동 시 데이터가 유지되도록 한다. '뒤로가기' 버튼은 이전 단계 데이터를 복원하고, 최종 제출 전까지 데이터를 보존한다. 다단계 폼은 현재 스텝 인덱스를 URL 파라미터 또는 상태로 관리하고, 편집 모드에서도 동일한 스텝 전진/후진 로직을 사용한다. 모든 스텝 전환 조건은 단위 테스트로 커버한다."

### Pattern: 숫자 입력 필드 음수/비숫자 허용

### Pattern: i18n 키 누락으로 인한 raw 키 노출
- **빈도**: 1회 (낮음)
- **증상**: 로그인 버튼이 한국어 대신 영어나 i18n 키 문자열(예: `auth.landing.loggingIn`)로 잠깐 표시됨
- **근본 원인**: 코드에서 i18n 키를 참조하지만 번역 파일(ko.json, en.json)에 해당 키가 정의되지 않음
- **영향 화면**: 로그인 페이지 버튼
- **예방 스펙**: "[클라이언트] i18n 키를 새로 추가하거나 참조할 때 ko.json과 en.json에 동시에 등록한다. 빌드 또는 CI 단계에서 누락된 i18n 키를 감지하는 린트 규칙을 적용한다."
- **사례**: [#1515]

## Category: 카메라/권한

## Category: AI 생성 / 디자인 도구

### Pattern: AI 생성 HTML 렌더링 오류 (스크립트 파싱 깨짐)
- **빈도**: 2회 (낮음)
- **증상**: 디자인 뷰어에 생성된 디자인 대신 raw HTML/코드가 표시되거나 빈 화면이 나타남
- **근본 원인**: AI가 생성한 `<script>` 태그 내부에 `</script>` 문자열이 포함되어 브라우저 파서가 스크립트 블록을 조기 종료, 이후 스크립트가 텍스트로 렌더링됨
- **영향 화면**: 디자인 생성 미리보기, 디자인 편집 뷰
- **예방 스펙**: "[서버/클라이언트] AI 생성 HTML을 iframe에 삽입하기 전 인라인 `<script>` 블록 내의 `</script>` 문자열을 이스케이프 처리하는 sanitizer를 반드시 적용한다. 생성된 HTML에 닫는 태그(`</body></html>`)가 없을 경우 자동 추가한다."
- **사례**: [#3044, #3107]

### Pattern: 버전 히스토리 데이터 손실
- **빈도**: 2회 (낮음)
- **증상**: 페이지 새로고침 후 버전 히스토리가 사라지거나, 버전 복원 후 다시 새로고침하면 원래 버전으로 돌아옴
- **근본 원인**: 버전 히스토리를 Redux 클라이언트 상태에만 저장하고 백엔드에 영속화하지 않음. 복원 API 호출도 미구현
- **영향 화면**: 디자인 편집 버전 히스토리
- **예방 스펙**: "[서버] 디자인 편집 메시지에 htmlSnapshot 필드를 저장하여 히스토리를 영속화한다. [클라이언트] 페이지 로드 시 편집 메시지의 htmlSnapshot으로 버전 히스토리를 복원하고, 버전 복원 시 반드시 백엔드 API를 호출해 서버 상태를 갱신한다."
- **사례**: [#3094, #3043]

## Category: 커뮤니티/게시판

- **영향 화면**: 커뮤니티 목록, 커뮤니티 상세
- **예방 스펙**: "[클라이언트] 이미지 렌더링 시 URL 유효성을 확인하고 실패 시 대체 이미지를 표시한다. 비로그인 사용자도 공개 게시글을 열람할 수 있도록 하고, 댓글/좋아요 등 인터랙션 시점에만 로그인을 요구한다. [서버] 파일 첨부 업데이트 API는 기존 파일 삭제 후 새 파일 저장을 트랜잭션으로 처리한다."

## Category: API/서버

### Pattern: API 500 에러 및 서버 예외 처리 미흡
- **빈도**: 2회 (낮음)
- **증상**: PUT /user/{id} 엔드포인트에서 최소 업데이트 요청에도 500 반환. 비밀번호 변경 성공 후 404 에러
- **근본 원인**: 컨트롤러에서 null 체크 또는 관계 처리 누락. 성공 응답 코드 오류
- **영향 화면**: 사용자 프로필 수정, 비밀번호 변경
- **예방 스펙**: "[서버] 모든 UPDATE 엔드포인트는 partial update를 지원하고 null/undefined 필드를 무시한다. 컨트롤러에서 DB 쿼리 전 입력값 유효성을 검사하고 적절한 HTTP 상태 코드(200, 400, 404, 500)를 반환한다. 전역 예외 핸들러를 구현하여 미처리 예외가 500으로 반환되지 않도록 한다."
- **사례**: [#1539, #1272]

## Category: OTP/이메일 인증

### Pattern: OTP 타이머 및 에러 상태 불일치
- **빈도**: 3회 (중간)
- **증상**: OTP 입력 후 4분이 지났는데도 에러가 지속 표시됨. OTP 유효 시간이 재발송 타이머보다 짧아 재발송 전에 코드가 만료됨. 만료된 인증 코드에 대한 오류 문구가 부정확함.
- **근본 원인**: 서버의 OTP 만료 시간(TTL)과 클라이언트 카운트다운 타이머 값이 동기화되지 않음. 에러 상태를 OTP 입력 시도 시점에만 클리어하고 타이머 만료 시에는 클리어하지 않음.
- **영향 화면**: OTP 인증 화면, 이메일 인증 화면
- **예방 스펙**: "[서버] OTP 유효 시간(TTL)을 API 응답에 포함하여 클라이언트에 전달한다. 재발송 대기 시간은 반드시 OTP 유효 시간보다 짧거나 같아야 한다. [클라이언트] OTP 타이머는 서버 응답의 TTL 값으로 초기화하고, 타이머 만료 시 에러 메시지를 자동 초기화하여 '코드가 만료되었습니다' 안내로 전환한다. 재발송 버튼 활성화 조건과 OTP 만료 조건을 동일한 TTL 기준으로 통일한다."
- **사례**: [Trustix #2032, Trustix #2021, Trustix #2024]

## Category: 데이터 무결성/연동

### Pattern: 공유 데이터 의도치 않은 연동 변경
- **빈도**: 4회 (중간)
- **증상**: 한 화면에서 회원 이름을 수정하면 다른 화면(티켓 관리)에서도 변경됨. 매치에 재할당 시 이전에 발급된 QR 링크와 데이터가 초기화됨. 다중 회원·다중 티켓 할당 시 이전 입력 데이터가 삭제됨. 업로드한 25개 파일 중 14개만 시스템에 반영됨.
- **근본 원인**: 서로 다른 엔티티(회원 프로필, 티켓, QR코드)가 참조가 아닌 값 복사로 연결되어 한쪽 수정이 다른 쪽에 전파됨. 재할당 로직이 기존 연관 데이터를 초기화하는 DELETE+INSERT 방식으로 구현됨. 대량 업로드 시 중간 실패 항목 롤백 로직 미흡.
- **영향 화면**: 회원 관리, 티켓 할당, QR 관리, 파일 업로드
- **예방 스펙**: "[서버] 엔티티 간 연결은 값 복사가 아닌 외래키 참조(FK)로 구현한다. 재할당(unassign→reassign) 시 기존 연관 데이터(QR 코드, 이전 설정)를 보존하고 변경된 항목만 업데이트하는 PATCH 방식을 사용한다. 대량 데이터 업로드는 트랜잭션으로 처리하고, 일부 실패 시 성공 건수와 실패 건수를 명시적으로 응답에 포함한다. [클라이언트] 다중 선택 입력 시 이전 선택값을 유지하고 새 항목만 추가하는 방식으로 상태를 관리한다."
- **사례**: [Trustix #2029, Trustix #2028, Trustix #1975, Trustix #1981]

## Category: 포인트/크레딧 관리

### Pattern: 포인트/크레딧 잔액 계산 오류 및 미갱신
- **빈도**: 3회 (중간)
- **증상**: 포인트를 10 충전 후 2 사용 시 잔액이 8이어야 하는데 개요 화면에서는 2, 포인트 관리 화면에서는 9로 다르게 표시됨. 포인트 추가 버튼이 동작하지 않음. 트랜잭션 히스토리가 업데이트되지 않음.
- **근본 원인**: 포인트 잔액을 여러 곳(개요 집계 쿼리, 포인트 관리 상세 쿼리)에서 각각 계산하여 동기화 오류 발생. 충전과 차감 이벤트를 별도 트랜잭션으로 처리하지 않아 중간 상태가 노출됨.
- **영향 화면**: 포인트 개요, 포인트 관리, 트랜잭션 히스토리
- **예방 스펙**: "[서버] 포인트 잔액은 단일 소스(포인트 테이블의 current_balance 컬럼)에서만 조회하고, 충전·차감 시 트랜잭션으로 원자성을 보장한다. 개요 API와 포인트 상세 API가 동일한 잔액 계산 로직을 사용하도록 공통 서비스로 분리한다. [클라이언트] 포인트 변경 후 개요와 포인트 관리 화면 양쪽의 쿼리 캐시를 동시에 invalidate한다."
- **사례**: [Trustix #2035, Trustix #1970, Trustix #2001]

## Category: 데이터 정렬/구분

### Pattern: 시간 기반 항목 분류/정렬 미적용
- **빈도**: 5회 (높음)
- **증상**: 지난 매치가 할당 대상 목록에 계속 표시됨. 진행 중/예정/종료 매치가 정해진 순서 없이 혼재됨. 날짜/시간 기준 정렬이 올바르게 적용되지 않음. 티켓 관리 화면에 '예정 매치'와 '지난 매치' 구분 섹션이 없음.
- **근본 원인**: 목록 조회 API에서 현재 시간 기준 필터링(status=ongoing/upcoming/past) 로직이 미구현되거나 날짜 비교 조건이 누락됨. UI에서 시간 기반 탭/섹션 분리 없이 전체 목록을 단순 나열.
- **영향 화면**: 매치 할당 화면, 매치 목록, 티켓 관리 화면
- **예방 스펙**: "[서버] 매치 목록 API는 현재 시간을 기준으로 upcoming(시작 전), ongoing(진행 중), past(종료) 상태를 자동으로 분류하여 반환한다. 할당 대상 매치 API는 기본적으로 미래/진행 중 매치만 포함한다. [클라이언트] 매치 목록 및 티켓 관리 화면은 '예정/진행 중'과 '지난 매치'를 별도 탭 또는 섹션으로 구분하여 표시한다. 기본 정렬은 날짜 오름차순(가장 가까운 예정이 상단)으로 설정한다."
- **사례**: [Trustix #2017, Trustix #1989, Trustix #1985, Trustix #2030, Trustix #1988]

## Category: 역할/권한 관리 (RBAC)

### Pattern: 사용자 역할 정의 불완전 및 역할 외 액션 허용
- **빈도**: 3회 (중간)
- **증상**: 시스템에 허용되지 않은 역할로 등록 가능. 관리자가 공급자 정보를 편집할 수 있어야 하나 일부만 편집 가능. 봇이 신청서를 반복 제출하는 스팸 방지 미흡.
- **근본 원인**: 허용 역할 목록이 서버와 클라이언트에서 다르게 정의되거나 Enum이 단일 소스로 관리되지 않음. 역할별 허용 액션 매트릭스가 명세화되지 않아 구현 누락. 폼 제출에 봇 방지(CAPTCHA, rate limiting) 미적용.
- **영향 화면**: 회원가입 화면, 관리자 사용자 관리, 공급자 관리
- **예방 스펙**: "[서버] 허용 역할 Enum을 단일 상수 파일로 정의하고 DB 컬럼 CHECK 제약으로 강제한다. 각 API 엔드포인트에 허용 역할 목록을 데코레이터로 명시한다. 공개 폼 제출 API에 rate limiting(동일 IP 분당 최대 N회)을 적용한다. [클라이언트] 회원가입 폼에 reCAPTCHA 또는 Honeypot 필드를 적용하여 봇 제출을 방지한다."
- **사례**: [Activity Coaching #2899, Ktalk #1021, Trustix #1997]

## Category: 하드코딩/동적 데이터 미연결

### Pattern: 동적 데이터가 하드코딩으로 고정
- **빈도**: 4회 (중간)
- **증상**: "Premier League" 텍스트가 게임 정보에서 동적으로 불러와야 하는데 하드코딩됨. 경쟁 로고가 데이터베이스가 아닌 정적 파일로 관리됨. 표시 옵션이 잘못된 레이블로 고정됨. 통화 기호가 파운드(£) 대신 달러($)로 하드코딩됨.
- **근본 원인**: 초기 개발 시 임시 하드코딩한 값을 실제 데이터 연결로 교체하지 않음. 다국가/다통화 지원을 고려하지 않고 특정 값으로 고정.
- **영향 화면**: 티켓 화면, 경쟁 목록, 매치 관리 화면
- **예방 스펙**: "[클라이언트] 리그명, 팀명, 통화 기호 등 동적으로 변경될 수 있는 모든 텍스트는 API 응답 데이터에서 바인딩한다. 하드코딩된 문자열은 코드 리뷰 체크리스트 항목으로 포함하여 PR 시 검출한다. [서버] 경쟁/리그 정보는 별도 테이블로 관리하고 CRUD API를 제공한다. 통화는 설정값(config)에서 로드하며 기본값은 프로젝트 요구사항에 맞게 설정한다."
- **사례**: [Trustix #2006, Trustix #2015, Trustix #1984, Trustix #1987]

## Category: UI 명칭/문구 일관성

### Pattern: 화면 간 명칭/ID 불일치
- **빈도**: 5회 (높음)
- **증상**: 'Member ID'와 'Member No.'가 화면에 따라 혼재. '회원 관리'와 '티켓 관리' 화면에서 동일 항목의 이름이 다름. 할당 화면에서 회원 번호만 표시되고 이름이 표시되지 않음. 필드 순서가 화면마다 달라 사용자 혼란 유발.
- **근본 원인**: 초기 스펙에서 용어 정의 없이 각 화면 개발자가 독립적으로 레이블을 정의. 공통 레이블 상수 미사용. 디자인 QA 시 명칭 일관성 미검토.
- **영향 화면**: 회원 관리, 티켓 관리, 할당 화면, 전체 관리자 페이지
- **예방 스펙**: "[클라이언트] 도메인 객체의 모든 레이블(Member ID/No., 필드명, 상태명 등)은 단일 i18n 또는 상수 파일에서 관리하고 모든 화면이 이를 참조한다. [서버] API 응답 필드명을 스펙 문서와 동일하게 맞추고, 필드명 변경 시 클라이언트와 함께 일괄 반영한다. PRD 작성 시 도메인 용어 사전(Glossary)을 정의하고 개발 전에 확정한다."
- **사례**: [Trustix #2018, Trustix #2014, Trustix #2019, Trustix #2011, Trustix #2016]

## Category: 선물/아이템 중복 처리

### Pattern: 중복 액션으로 인한 아이템/선물 중복 지급
- **빈도**: 3회 (중간)
- **증상**: 비디오 연장 중 같은 선물이 두 번 전송됨. 선물 선택 UI에서 이미 선택한 항목이 다시 선택 가능한 상태로 표시됨. 이벤트가 두 번 발생하여 중복 처리됨.
- **근본 원인**: 선물 전송 이벤트 리스너가 중복 등록되거나, 소켓 이벤트 수신 시 멱등성 처리 없이 매번 새 선물을 생성. 선택 상태가 UI와 서버 간에 동기화되지 않음.
- **영향 화면**: 비디오 통화 선물 UI, 아이템 선택 화면
- **예방 스펙**: "[서버] 선물 전송 API는 idempotency key(예: session_id + gift_id + timestamp 조합)를 사용하여 동일 요청의 중복 처리를 방지한다. [클라이언트] 선물/아이템 전송 버튼은 클릭 후 서버 응답 수신 전까지 비활성화한다. 소켓 이벤트 리스너는 컴포넌트 마운트 시 1회만 등록하고 언마운트 시 반드시 해제한다."
- **사례**: [Blink #2983, Blink #3005, Bside #3089]

## Category: 아트워크/컨텐츠 캡처

### Pattern: 컨텐츠 캡처 후 컬렉션 미분류
- **빈도**: 3회 (중간)
- **증상**: 사진 캡처 후 캡처 컬렉션이 생성되지 않음. 캡처된 아트워크가 올바른 컬렉션에 정렬되지 않음. 사진 캡처 후 아트워크 추론 API가 호출되지 않음.
- **근본 원인**: 캡처 완료 이벤트 후 컬렉션 생성 API 호출 로직이 누락됨. AI 추론 API 트리거가 캡처 완료 핸들러에 연결되지 않음.
- **영향 화면**: 아트워크 캡처 화면, 컬렉션 목록
- **예방 스펙**: "[클라이언트] 사진 캡처 완료 시 (1) 아트워크 추론 API 호출 → (2) 추론 결과 기반 컬렉션 자동 분류 → (3) 컬렉션 목록 갱신의 순서로 플로우를 구현한다. 각 단계 실패 시 사용자에게 적절한 에러 메시지를 표시하고 수동 컬렉션 선택 옵션을 제공한다."
- **사례**: [Bside #3008, Bside #3090, Bside #3089]

## Category: 운동/처방 관리

### Pattern: 운동 타이머 컨텍스트별 동작 분기 미구현
- **빈도**: 2회 (낮음)
- **증상**: 새 운동 시작 시 타이머가 초기화되지 않고 이전 운동의 시간이 이어짐. 이전에 진행 중이던 운동으로 돌아올 때 저장된 시간부터 재개되지 않음.
- **근본 원인**: 운동 타이머 상태가 운동 ID와 독립적으로 전역 관리되어 운동 전환 시 리셋되지 않음.
- **영향 화면**: 운동 실행 화면, 운동 목록
- **예방 스펙**: "[클라이언트] 운동 타이머는 운동 ID를 키로 하는 맵 구조로 관리한다. 신규 운동 시작 시 해당 운동 ID에 대한 타이머를 0에서 시작하고, 이전 운동으로 복귀 시 저장된 경과 시간에서 재개한다. [서버] 운동 진행 상태(경과 시간, 완료 여부)를 운동 ID별로 저장하고, 앱 재시작 시 복원 가능하도록 API를 제공한다."
- **사례**: [Activity Coaching #2917, Activity Coaching #2898]

### Pattern: 개요/대시보드 정적 데이터 표시
- **빈도**: 4회 (중간)
- **증상**: 개요 페이지의 수치(티켓 수, 매치 수, 포인트 등)가 새로운 데이터 추가 후에도 갱신되지 않고 정적으로 표시됨. 최근 티켓 옵션이 동작하지 않음.
- **근본 원인**: 개요/대시보드 페이지의 집계 데이터를 초기 로드 시 1회만 조회하고, 이후 데이터 변경 시 재조회 로직이 없음.
- **영향 화면**: 관리자 개요 페이지, 공급자 사이드 최근 활동 패널
- **예방 스펙**: "[클라이언트] 대시보드/개요 페이지의 집계 수치는 화면 포커스(useFocusEffect) 또는 주기적 폴링(30초)으로 자동 갱신한다. 매치/티켓/포인트 생성·수정 이벤트 발생 시 관련 쿼리 키를 즉시 invalidate하여 개요 데이터를 갱신한다."
- **사례**: [Trustix #1994, Trustix #1976, Trustix #1992, Trustix #1977]

### Pattern: 관리자 필드 편집 불가 (비활성화 미구현)
- **빈도**: 3회 (중간)
- **증상**: 관리자가 티켓 상세 항목을 수정하려 해도 필드가 편집 불가 상태. 팀 관리 페이지에 편집 기능 자체가 없음.
- **근본 원인**: 관리자 전용 편집 권한이 있음에도 입력 필드가 disabled 또는 readOnly로 고정되거나 편집 모드 전환 로직이 미구현됨.
- **영향 화면**: 관리자 티켓 상세, 관리자 팀 관리, 공급자 정보 편집
- **예방 스펙**: "[클라이언트] 관리자 권한 사용자는 상세 페이지에서 '편집' 버튼 클릭 시 관련 필드가 편집 가능 상태로 전환되어야 한다. 역할(role)에 따라 편집 가능 필드 목록을 별도 상수로 정의한다. [서버] 관리자 편집 API의 허용 필드 목록(UpdateDto)이 UI에서 제공하는 편집 필드와 일치하는지 PR 시 체크리스트로 검증한다."
- **사례**: [Trustix #2013, Trustix #2012, Trustix #1997]

### Pattern: 클릭 가능 영역 불일치 (레이블 미동작)
- **빈도**: 1회 (낮음)
- **증상**: 체크박스 옆 텍스트 레이블 영역을 클릭해도 체크박스가 토글되지 않고, 체크박스 아이콘 자체만 클릭 가능.
- **근본 원인**: label 태그로 체크박스를 감싸지 않거나 htmlFor/id 속성이 연결되지 않음.
- **영향 화면**: 체크박스가 포함된 폼, 동의 화면, 선택 목록
- **예방 스펙**: "[클라이언트] 체크박스, 라디오버튼 등 선택형 입력 요소는 반드시 label htmlFor={id} 또는 label 안에 input을 넣는 형태로 구현하여 레이블 전체 영역이 클릭 가능하도록 한다."
- **사례**: [Blink #1582]
