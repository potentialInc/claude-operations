## 카테고리: 인증/로그인

### Pattern: 회원가입 폼 — API 미연결
- **빈도**: 2회 (낮음)
- **증상**: 회원가입 폼 제출 시 계정이 생성되지 않거나, 특정 필드(라이선스 번호, 병원 코드)가 저장되지 않음
- **근본 원인**: handleSubmit에서 실제 API 호출 없이 navigate()만 실행하거나, 백엔드 DTO/Entity에 필드가 누락되어 전체 파이프라인이 불완전
- **영향 화면**: 코치 회원가입 폼
- **예방 스펙**: "[클라이언트] 회원가입 폼 제출 핸들러는 반드시 인증 API를 호출해야 하며, 로딩 상태·에러 핸들링·성공 메시지 UI를 포함해야 한다. [서버] RegisterDto, Entity, Service 세 레이어 모두에 동일 필드가 존재하는지 PR 체크리스트로 검증한다."
- **사례**: [#3070, #3071]

### Pattern: 로그인 실패 메시지 미표시
- **빈도**: 1회 (낮음)
- **증상**: 잘못된 비밀번호 입력 시 아무런 오류 메시지가 나타나지 않음
- **근본 원인**: 로그인 실패 시 에러 상태를 UI에 바인딩하지 않음
- **영향 화면**: 관리자 로그인 페이지
- **예방 스펙**: "[클라이언트] 로그인 실패 응답(401/403) 수신 시 반드시 인라인 오류 메시지를 노출해야 한다. 에러 메시지는 i18n 키를 통해 다국어를 지원한다."
- **사례**: [#1531]

### Pattern: 신규 가입 후 로그인 루프
- **빈도**: 1회 (낮음)
- **증상**: 신규 가입 직후 로그인 시 등록 페이지로 계속 리다이렉트되는 무한 루프
- **근본 원인**: 가입 완료 후 인증 상태가 갱신되지 않아 라우팅 가드가 미인증으로 판단하고 등록 페이지로 반복 전송
- **영향 화면**: 신규 사용자 로그인 플로우
- **예방 스펙**: "[클라이언트] 회원가입 성공 후 인증 토큰/상태를 즉시 갱신하고, 라우팅 가드가 새 상태를 참조하도록 한다. 가입→로그인→홈 플로우를 E2E 테스트로 커버한다."
- **사례**: [#1283]

### Pattern: 소셜 로그인 플로우 미최적화
- **빈도**: 1회 (낮음)
- **증상**: 소셜 로그인 플로우가 불필요한 단계를 거치거나 UX가 불연속적
- **근본 원인**: OAuth 콜백 처리 후 세션 복원 및 리다이렉트 로직 미비
- **영향 화면**: 소셜 로그인 페이지
- **예방 스펙**: "[서버] OAuth 콜백 핸들러는 토큰 발급 후 즉시 클라이언트 딥링크로 리다이렉트해야 한다. [클라이언트] 소셜 로그인 완료 후 이전 페이지 또는 홈으로 복귀해야 한다."
- **사례**: [#3082]

## 카테고리: 이미지/파일 업로드

### Pattern: 업로드된 이미지 채팅 UI 미표시
- **빈도**: 1회 (낮음)
- **증상**: 이미지를 전송했음에도 채팅 UI에 이미지가 표시되지 않음
- **근본 원인**: 이미지 데이터 URL이 백엔드로는 전송되나 Redux 상태 및 MessageBubble 컴포넌트에 바인딩되지 않음
- **영향 화면**: 채팅 상세 페이지 (이미지 첨부)
- **예방 스펙**: "[클라이언트] 파일/이미지 전송 후 해당 메시지 객체에 미리보기 URL을 포함해 즉시 렌더링해야 한다. 업로드 성공 여부와 관계없이 낙관적 UI로 먼저 표시한다."
- **사례**: [#3081]

### Pattern: STL 파일 삭제 후 업로드 흐름 중단
- **빈도**: 1회 (낮음)
- **증상**: 업로드된 STL 파일 삭제 시 "Skip for now" 버튼이 동작하지 않아 재업로드 없이 진행 불가
- **근본 원인**: 파일 삭제 후 업로드 상태 초기화가 완전하지 않아 다음 단계 진행 조건이 잘못 평가됨
- **영향 화면**: STL 파일 업로드, 주문 편집 페이지
- **예방 스펙**: "[클라이언트] 파일 삭제 시 업로드 상태를 완전히 초기화(null/empty)하고, 필수 파일 없이도 '건너뛰기'가 가능한 경우 해당 조건을 명시적으로 처리한다."
- **사례**: [#1279]

## 카테고리: 리스트/페이지네이션

### Pattern: 관리자 테이블 Export 버튼 미동작
- **빈도**: 2회 (낮음)
- **증상**: 관리자 환자/코치 테이블에서 Export 버튼을 클릭해도 아무 일도 일어나지 않음
- **근본 원인**: Export 버튼에 클릭 핸들러 또는 실제 데이터 다운로드 로직이 구현되지 않음
- **영향 화면**: 관리자 환자 테이블, 관리자 코치 테이블
- **예방 스펙**: "[클라이언트] 버튼 컴포넌트에 onClick 핸들러가 없으면 비활성(disabled) 상태로 표시하거나, 구현 전까지 버튼 자체를 숨겨야 한다. Export 기능은 CSV/Excel 다운로드 API 연결 후 활성화한다."
- **사례**: [#3073, #3072]

### Pattern: 설문 히스토리 페이지네이션 누락
- **빈도**: 1회 (낮음)
- **증상**: 설문 히스토리가 10개만 표시되고 더 이상 로드되지 않음
- **근본 원인**: useHistory 훅이 페이지네이션 파라미터를 지원하지 않고 고정 limit=10으로 조회
- **영향 화면**: 환자 설문 히스토리 탭
- **예방 스펙**: "[클라이언트] 리스트 API 연동 시 pagination 파라미터(page, limit)를 지원하고 'Load More' 또는 무한 스크롤을 구현한다. [서버] 리스트 API는 반드시 total, page, limit을 포함한 페이지네이션 응답 구조를 반환한다."
- **사례**: [#1530]

### Pattern: 주문 상세 테이블 주문 미표시
- **빈도**: 1회 (낮음)
- **증상**: 사용자 상세 페이지의 주문 정보 테이블에 일부 주문이 표시되지 않음 (카운트는 2이나 1개만 노출)
- **근본 원인**: 페이지네이션·필터 적용 문제 또는 탭 조건(Confirmed vs Draft) 불일치로 인한 데이터 누락
- **영향 화면**: 관리자 사용자 상세 → 주문 정보 섹션
- **예방 스펙**: "[서버] 사용자 주문 목록 API는 상태 필터 없이 전체 주문 수(totalCount)를 별도로 반환해야 한다. [클라이언트] 테이블 상단 카운트와 실제 렌더링 행 수가 불일치하면 경고 로그를 출력한다."
- **사례**: [#1542]

## 카테고리: 실시간 데이터 (채팅/소켓)

### Pattern: 채팅 메시지 정렬 오류
- **빈도**: 1회 (낮음)
- **증상**: 채팅 메시지가 최신 순(내림차순)으로 표시되어 가장 최근 메시지가 상단에 위치
- **근본 원인**: 백엔드가 페이지네이션을 위해 DESC 정렬로 반환하지만 프론트엔드가 배열을 역순으로 정렬하지 않고 그대로 렌더링
- **영향 화면**: 환자 채팅 상세, 코치 채팅 상세
- **예방 스펙**: "[서버] 채팅 메시지 목록 API는 실제 렌더링 순서(오래된 순 = ASC)를 기본으로 반환하거나, DESC 정렬 시 응답에 명시한다. [클라이언트] 메시지 렌더링 전 항상 createdAt ASC 기준으로 정렬 후 표시한다."
- **사례**: [#2977]

### Pattern: 채팅 발신자 라벨 오인식
- **빈도**: 1회 (낮음)
- **증상**: 관리자가 대시보드에서 보낸 메시지가 채팅에서 'Coach'로 표시됨
- **근본 원인**: 발신자 식별을 ID 비교에만 의존하여 관리자가 코치이기도 한 경우 역할 오인식
- **영향 화면**: 환자 채팅 상세 페이지
- **예방 스펙**: "[서버] 소켓 메시지 브로드캐스트 및 채팅방 조회 응답에 sender.role 필드를 포함해야 한다. [클라이언트] 발신자 표시는 ID 비교보다 role 필드를 우선 사용한다."
- **사례**: [#2978]

### Pattern: 채팅 중복 채팅방 생성
- **빈도**: 1회 (낮음)
- **증상**: 환자와 코치 사이에 동일한 채팅방이 중복 생성되어 두 개의 채팅방이 표시됨
- **근본 원인**: 채팅방 생성 API가 getOrCreate 패턴이 아닌 매번 새 레코드를 삽입하여 레이스 컨디션 발생
- **영향 화면**: 채팅방 목록
- **예방 스펙**: "[서버] 채팅방 생성 엔드포인트는 getOrCreate 패턴을 사용하고, DB UNIQUE 제약 + 레이스 컨디션 처리(23505 에러 캐치)를 구현해야 한다. 중복 채팅방이 존재할 경우 마이그레이션으로 병합한다."
- **사례**: [#2337]

### Pattern: 실시간 상태/알림 미동기화
- **빈도**: 1회 (낮음)
- **증상**: 라이브 서버에서 주문 상태 변경 및 알림이 실시간으로 반영되지 않음
- **근본 원인**: 상태 업데이트와 알림 이벤트가 WebSocket/Server-Sent Events로 전파되지 않음
- **영향 화면**: 주문 관리 대시보드, 알림 센터
- **예방 스펙**: "[서버] 주문 상태 변경 시 관련 사용자에게 실시간 이벤트를 emit해야 한다. [클라이언트] 소켓 연결 해제 시 재연결 및 놓친 이벤트 재조회 로직을 구현한다."
- **사례**: [#1362]

## 카테고리: 데이터 동기화 / 타임존

### Pattern: 타임존 불일치로 인한 날짜 오프셋
- **빈도**: 3회 (중간)
- **증상**: 날짜 선택 시 하루 전/후 날짜가 선택되거나, 오늘의 운동·설문 데이터가 0으로 표시되거나, 캘린더 점 표시가 실제 데이터와 일치하지 않음
- **근본 원인**: `new Date('YYYY-MM-DD')` 또는 `toISOString()`이 UTC 기준으로 변환되어 KST(UTC+9) 환경에서 날짜가 1일 shift. `new Date()` 를 매 렌더에서 생성하는 것도 불필요한 재계산 유발
- **영향 화면**: 환자 홈(캘린더), 운동 로그, 설문 상태 표시
- **예방 스펙**: "[클라이언트] 날짜 문자열(YYYY-MM-DD)을 Date 객체로 변환할 때 반드시 로컬 타임존 기준 파싱(로컬 자정)을 사용하고 toISOString() 대신 직접 문자열 포맷(YYYY-MM-DD)을 유지한다. today 기준 날짜는 useMemo로 안정화한다. [서버] 날짜 범위 조회 시 start-of-day(00:00:00.000) ~ end-of-day(23:59:59.999) 정규화를 DB 쿼리에 적용한다."
- **사례**: [#2975, #2993, #3001]

### Pattern: 캘린더 점 표시와 실제 데이터 불일치
- **빈도**: 2회 (낮음)
- **증상**: 설문 제출 후 캘린더 점이 갱신되지 않거나, 날짜 파싱 오류로 점 위치가 잘못 표시됨
- **근본 원인**: (1) 설문 제출 후 쿼리 캐시 무효화 누락, (2) new Date() 기반 날짜 파싱의 타임존 오류
- **영향 화면**: 환자 홈 캘린더 월간 뷰
- **예방 스펙**: "[클라이언트] 데이터 제출 성공 후 관련 쿼리 키를 즉시 invalidate해야 한다. 날짜 비교는 문자열 기반(YYYY-MM-DD) 방식을 사용한다."
- **사례**: [#3001, #2997]

### Pattern: 운동 카운트/완료 상태 불일치
- **빈도**: 2회 (낮음)
- **증상**: 운동 완료 수 또는 처방 수가 홈 화면에서 0으로 표시되거나 오래된 캐시 값을 반영
- **근본 원인**: (1) getDaySummary가 ExerciseDay 레코드가 없으면 null 반환하여 0/0 표시, (2) countActiveForPatient가 타임존 오류로 잘못된 날짜 조회, (3) 처방 비활성화 시 totalCount 검증 로직 미적용
- **영향 화면**: 환자 홈 운동 카드, 운동 기록 페이지
- **예방 스펙**: "[서버] getDaySummary는 ExerciseDay 레코드 부재 시 null 대신 현재 활성 처방 기반 synthetic summary를 반환해야 한다. 처방 변경 시 totalCount를 실시간 재검증한다."
- **사례**: [#3021, #2993]

## 카테고리: 관리자 대시보드

### Pattern: 관리자 사용자 업데이트 DTO 필드 불일치
- **빈도**: 2회 (낮음)
- **증상**: 관리자가 환자 정보(fullName, phone, birth) 또는 코치 배정(coachId)을 저장하려 하면 400 오류 발생
- **근본 원인**: 프론트엔드가 전송하는 필드가 UpdateUserDto의 허용 필드에 없어 ValidationPipe(forbidNonWhitelisted)가 요청을 거부
- **영향 화면**: 관리자 환자 상세, 사용자 관리
- **예방 스펙**: "[서버] UpdateUserDto는 클라이언트가 전송 가능한 모든 수정 필드를 @IsOptional()로 포함해야 한다. DTO 변경 시 프론트엔드 타입 정의도 동시에 업데이트한다."
- **사례**: [#2940, #2921]

### Pattern: 관리자 통계 날짜 계산 오류
- **빈도**: 1회 (낮음)
- **증상**: 신규 사용자의 경과 일수(daysElapsed)가 음수로 표시됨
- **근본 원인**: registrationDate가 자정으로 초기화되지 않아 같은 날 등록한 사용자의 경과 일수가 음수
- **영향 화면**: 관리자 환자 목록, 통계 페이지
- **예방 스펙**: "[서버] 날짜 차이 계산 시 두 Date 객체 모두 `setHours(0,0,0,0)`으로 자정 정규화 후 연산한다."
- **사례**: [#2921]

## 카테고리: 폼/입력

### Pattern: 숫자 입력 필드 음수/비숫자 허용
- **빈도**: 2회 (낮음)
- **증상**: 운동 세트 수, 보행 시간, 걸음 수 등 수치 입력 필드에 음수나 문자(e, +, -)가 입력됨
- **근본 원인**: `<input type="number">` 의 기본 동작이 e, +, - 키를 허용하고, 클라이언트 측 keydown 차단 로직이 미구현
- **영향 화면**: 설문 입력 폼, 운동 처방 입력 폼
- **예방 스펙**: "[클라이언트] 숫자 전용 입력 필드에 onKeyDown 핸들러로 e, E, +, -, . 키를 preventDefault 처리한다. min=0 속성을 함께 적용하고, 폼 제출 시 서버에서도 양수 정수 범위를 재검증한다."
- **사례**: [#3023, #1526]

### Pattern: 비밀번호 확인 불일치 메시지 미표시
- **빈도**: 1회 (낮음)
- **증상**: 계정 생성 시 비밀번호와 확인 비밀번호가 다를 때 사용자에게 안내 메시지가 없음
- **근본 원인**: 비밀번호 확인 필드에 실시간 또는 제출 시 비교 유효성 검사 로직 미구현
- **영향 화면**: 회원가입, 계정 생성 모달
- **예방 스펙**: "[클라이언트] 비밀번호 확인 필드는 onChange 또는 onBlur 시 원래 비밀번호와 비교하여 불일치 시 즉시 인라인 경고 메시지를 표시한다. 제출 버튼은 일치할 때만 활성화한다."
- **사례**: [#3015]

### Pattern: 다단계 주문 폼 — 뒤로가기/편집 시 루프
- **빈도**: 2회 (낮음)
- **증상**: 주문 편집 중 뒤로가기 시 이미 지나온 페이지로 반복 이동하거나, 모든 치아를 크라운으로 선택 후 진행이 멈춤
- **근본 원인**: 다단계 폼의 스텝 상태 관리 로직이 편집 모드에서 조건을 잘못 평가하거나, 완료 조건이 초기 생성과 편집에서 다르게 처리됨
- **영향 화면**: 주문 생성/편집 폼 (Abutment, Crown 단계)
- **예방 스펙**: "[클라이언트] 다단계 폼은 현재 스텝 인덱스를 URL 파라미터 또는 상태로 관리하고, 편집 모드에서도 동일한 스텝 전진/후진 로직을 사용한다. 모든 스텝 전환 조건은 단위 테스트로 커버한다."
- **사례**: [#1323, #1282]

## 카테고리: 네비게이션/라우팅

### Pattern: 뒤로가기 버튼 잘못된 리다이렉션
- **빈도**: 3회 (중간)
- **증상**: 뒤로가기 버튼이 예상 페이지(주문 목록, 로그인 등) 대신 이전 방문 페이지나 엉뚱한 페이지로 이동
- **근본 원인**: 뒤로가기 버튼이 `navigate(-1)` 또는 `history.back()`에 의존하여 히스토리 스택 상태에 따라 목적지가 달라짐. 일부는 명시적 목적지 없이 구현됨
- **영향 화면**: 주문 상세, 세션 기록, 일반 페이지 내비게이션
- **예방 스펙**: "[클라이언트] 뒤로가기 버튼은 `navigate(-1)` 대신 명시적 목적지 경로(예: navigate('/orders'))를 사용한다. 페이지별 기본 뒤로가기 목적지를 스펙에 명시한다."
- **사례**: [#3054, #3025, #1324]

### Pattern: 페이지 리로드 시 첫 페이지로 리다이렉트
- **빈도**: 1회 (낮음)
- **증상**: 현재 페이지에서 새로고침 시 항상 첫 페이지(랜딩/홈)로 이동
- **근본 원인**: 라우팅이 인증 상태 복원 전에 실행되어, 토큰이 로컬스토리지에 있어도 미인증으로 판단하고 루트로 리다이렉트
- **영향 화면**: 모든 보호된 라우트
- **예방 스펙**: "[클라이언트] 보호된 라우트 평가는 인증 상태 복원이 완료된 후 실행해야 한다. 초기 로딩 중에는 빈 화면이나 스피너를 표시하고 라우트 리다이렉트를 지연한다."
- **사례**: [#3110]

### Pattern: 딥링크/앵커 내비게이션 오동작
- **빈도**: 1회 (낮음)
- **증상**: 프로토타입 내비게이션에서 인페이지 앵커(#section) 링크가 페이지 이동으로 처리되거나 공백 포함 링크가 매칭 실패
- **근본 원인**: 앵커 링크 처리 로직이 # 시작 URL을 catch-all 라우팅에서 예외 처리하지 않고, 링크의 공백을 URL 슬러그와 비교 시 정규화하지 않음
- **영향 화면**: 디자인 프리뷰 프로토타입 내비게이션
- **예방 스펙**: "[클라이언트] 라우팅 인터셉터에서 '#'으로 시작하는 href는 인페이지 스크롤로 처리하고 내비게이션에서 제외한다. 링크 매칭 시 공백을 하이픈으로 정규화하고 소문자로 통일한다."
- **사례**: [#3102]

## 카테고리: 성능 / 렌더링

### Pattern: 불필요한 리렌더링으로 인한 데이터 불안정
- **빈도**: 1회 (낮음)
- **증상**: 환자 홈 페이지에서 날짜 기반 스케줄 데이터가 매 렌더마다 재계산되어 UI 깜빡임 발생
- **근본 원인**: `new Date()`가 컴포넌트 렌더 함수 내에서 호출되어 매 렌더마다 새 객체 생성 → 의존 배열 불안정
- **영향 화면**: 환자 홈 페이지
- **예방 스펙**: "[클라이언트] 렌더 함수 내 `new Date()` 호출은 `useMemo(() => new Date(), [])` 또는 상수로 안정화한다. 날짜 의존 effect/memo의 deps 배열에 불안정한 객체를 포함하지 않는다."
- **사례**: [#2997]

### Pattern: 디자인 생성 후 페이지 리로드 필요
- **빈도**: 2회 (낮음)
- **증상**: 디자인 생성 완료 후 페이지를 새로고침해야 생성된 디자인이 표시됨
- **근본 원인**: SSE 생성 완료 핸들러에만 fetchDesign 로직이 있고 폴링, 폴백, safety-net 경로에는 재시도 로직이 없어 타이밍에 따라 데이터 로드 실패
- **영향 화면**: 디자인 생성 화면, 마이 디자인
- **예방 스펙**: "[클라이언트] 비동기 생성 완료 이벤트 수신 후 데이터 조회는 지수 백오프를 적용한 재시도 로직(최소 3회)을 모든 완료 경로(SSE, 폴링, 폴백)에 동일하게 적용한다."
- **사례**: [#3034, #3100]

## 카테고리: 검색/필터

### Pattern: 필터 초기화 시 전체 데이터 노출
- **빈도**: 1회 (낮음)
- **증상**: '전체 해제(Deselect All)' 필터 적용 시 빈 결과가 아닌 모든 데이터가 표시됨
- **근본 원인**: 선택 항목이 비어있을 때 필터 함수가 "필터 없음 = 전체 표시"로 처리하고, API 쿼리도 전체 데이터를 재요청
- **영향 화면**: 세션 캘린더 코치 필터
- **예방 스펙**: "[클라이언트] 필터 배열이 빈 경우 '아무것도 선택 안 됨 = 빈 결과' 와 '필터 없음 = 전체 표시'를 명확히 구분해야 한다. 필터 배열이 비어 있으면 API 요청을 skip하고 빈 배열을 반환한다."
- **사례**: [#3031]

### Pattern: 검색 기능 DB 미연결
- **빈도**: 1회 (낮음)
- **증상**: 검색창에 입력해도 결과가 변경되지 않음
- **근본 원인**: 검색 UI만 구현되고 실제 API 쿼리 파라미터 전달 로직이 연결되지 않음
- **영향 화면**: 검색 페이지
- **예방 스펙**: "[클라이언트] 검색 입력 컴포넌트는 debounce(300ms) 적용 후 검색어를 API 쿼리 파라미터로 전달해야 한다. 검색 결과 0건 시 빈 상태 UI를 표시한다."
- **사례**: [#3013]

## 카테고리: UI/레이아웃

### Pattern: 캘린더 이벤트 렌더링 오류 (블록/도트)
- **빈도**: 2회 (낮음)
- **증상**: 캘린더 월간 뷰에서 이벤트가 블록 대신 점으로 표시되거나, 주간/일간 뷰에서 인접한 이벤트가 겹쳐 보이고 배경 격자선이 비침
- **근본 원인**: FullCalendar 이벤트 객체에 `display: block` 속성 미설정, CSS 변수가 잘못된 형식(hsl() 래핑)으로 적용되어 배경색 렌더링 실패
- **영향 화면**: 관리자/코치 캘린더
- **예방 스펙**: "[클라이언트] FullCalendar 월간 뷰 이벤트는 `display: 'block'`을 이벤트 객체에 명시한다. Tailwind CSS v4에서 CSS 커스텀 속성은 `var(--color)` 형식을 사용하고 `hsl(var(--color))`로 감싸지 않는다."
- **사례**: [#3032, #2914]

### Pattern: i18n 키 누락으로 인한 raw 키 노출
- **빈도**: 1회 (낮음)
- **증상**: 로그인 버튼이 한국어 대신 영어나 i18n 키 문자열(예: `auth.landing.loggingIn`)로 잠깐 표시됨
- **근본 원인**: 코드에서 i18n 키를 참조하지만 번역 파일(ko.json, en.json)에 해당 키가 정의되지 않음
- **영향 화면**: 로그인 페이지 버튼
- **예방 스펙**: "[클라이언트] i18n 키를 새로 추가하거나 참조할 때 ko.json과 en.json에 동시에 등록한다. 빌드 또는 CI 단계에서 누락된 i18n 키를 감지하는 린트 규칙을 적용한다."
- **사례**: [#1515]

### Pattern: 프로필/편집 페이지 스크롤 불가
- **빈도**: 2회 (낮음)
- **증상**: 코치/환자 프로필 페이지의 콘텐츠가 넘칠 때 스크롤이 되지 않아 하단 내용을 볼 수 없음
- **근본 원인**: flex 레이아웃 컨테이너에 `min-h-0`이 없어 flex 아이템이 콘텐츠 높이 이하로 축소되지 않아 `overflow-y-auto`가 활성화되지 않음
- **영향 화면**: 코치 프로필, 환자 프로필
- **예방 스펙**: "[클라이언트] `overflow-y: auto`를 사용하는 flex 자식 요소에는 반드시 `min-h-0`을 함께 적용한다. 레이아웃 컴포넌트 패턴을 재사용할 때 해당 속성을 표준 템플릿에 포함한다."
- **사례**: [#2332, #3080]

## 카테고리: 결제/주문 관리

### Pattern: 주문 상태 표시 불일치
- **빈도**: 3회 (중간)
- **증상**: 결제 후 상태가 'Awaiting Production'으로 변경되지 않거나, 환불 요청/완료/거절 상태가 고객 트래킹 페이지에 반영되지 않음. 리비전 요청 상태가 대시보드에서 보이지 않음
- **근본 원인**: 상태 전환 로직 또는 클라이언트 표시 매핑에서 신규 상태값이 누락
- **영향 화면**: 주문 상세, 주문 트래킹, 관리자 대시보드
- **예방 스펙**: "[서버] 주문 상태 Enum을 단일 소스로 관리하고, 상태 전환 시 허용된 전환 경로만 허용하는 상태 머신을 구현한다. [클라이언트] 상태별 표시 레이블을 상수 맵으로 관리하고, 정의되지 않은 상태는 'Unknown' 등 폴백 표시를 한다."
- **사례**: [#1336, #1344, #1546]

### Pattern: 주문 폼 — 조건부 필드 표시 오류
- **빈도**: 4회 (중간)
- **증상**: 선택된 옵션에 따라 특정 필드가 표시/숨김되어야 하는데, 잘못된 조건으로 불필요한 필드가 노출되거나 필요한 필드가 숨겨짐 (Full Arch의 Retention Type, MUA 스캔바디 타입, FP1/FP2/FP3 옵션 등)
- **근본 원인**: 조건부 렌더링 로직의 조건식이 복잡한 주문 타입 조합을 올바르게 처리하지 못함
- **영향 화면**: 주문 생성/편집 폼 (Implant Full Arch, Crown, Bridge)
- **예방 스펙**: "[클라이언트] 주문 폼의 조건부 필드 표시 로직은 주문 타입별 결정 트리를 별도 상수/설정 파일로 분리하여 관리한다. 각 조합에 대한 표시/숨김 케이스를 테스트 케이스로 문서화한다."
- **사례**: [#1300, #1302, #1303, #1304]

### Pattern: 채팅 아이콘 미읽음 알림 미표시
- **빈도**: 2회 (낮음)
- **증상**: 새 채팅 메시지가 있어도 채팅 아이콘에 알림 뱃지가 표시되지 않음. 디자인 리뷰 코멘트 알림도 표시되지 않음
- **근본 원인**: 미읽음 메시지 카운트가 채팅 아이콘 컴포넌트에 전달되지 않거나, 알림 이벤트 구독이 누락
- **영향 화면**: 채팅 아이콘, 알림 센터
- **예방 스펙**: "[클라이언트] 채팅 아이콘은 미읽음 메시지 수를 실시간으로 반영하는 배지를 표시해야 한다. 소켓 수신 이벤트 또는 주기적 폴링으로 미읽음 수를 갱신한다."
- **사례**: [#1365, #1363]

## 카테고리: AI 생성 / 디자인 도구

### Pattern: AI 생성 HTML 렌더링 오류 (스크립트 파싱 깨짐)
- **빈도**: 2회 (낮음)
- **증상**: 디자인 뷰어에 생성된 디자인 대신 raw HTML/코드가 표시되거나 빈 화면이 나타남
- **근본 원인**: AI가 생성한 `<script>` 태그 내부에 `</script>` 문자열이 포함되어 브라우저 파서가 스크립트 블록을 조기 종료, 이후 스크립트가 텍스트로 렌더링됨
- **영향 화면**: 디자인 생성 미리보기, 디자인 편집 뷰
- **예방 스펙**: "[서버/클라이언트] AI 생성 HTML을 iframe에 삽입하기 전 인라인 `<script>` 블록 내의 `</script>` 문자열을 이스케이프 처리하는 sanitizer를 반드시 적용한다. 생성된 HTML에 닫는 태그(`</body></html>`)가 없을 경우 자동 추가한다."
- **사례**: [#3044, #3107]

### Pattern: 버전 히스토리 데이터 손실
- **빈도**: 2회 (낮음)
- **증상**: 페이지 새로고침 후 버전 히스토리가 사라지거나, 버전 복원 후 다시 새로고침하면 원래 버전으로 돌아옴
- **근본 원인**: 버전 히스토리를 Redux 클라이언트 상태에만 저장하고 백엔드에 영속화하지 않음. 복원 API 호출도 미구현
- **영향 화면**: 디자인 편집 버전 히스토리
- **예방 스펙**: "[서버] 디자인 편집 메시지에 htmlSnapshot 필드를 저장하여 히스토리를 영속화한다. [클라이언트] 페이지 로드 시 편집 메시지의 htmlSnapshot으로 버전 히스토리를 복원하고, 버전 복원 시 반드시 백엔드 API를 호출해 서버 상태를 갱신한다."
- **사례**: [#3094, #3043]
